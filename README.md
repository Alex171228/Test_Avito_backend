# Сервис бронирования переговорных комнат

[![Review Assignment Due Date](https://classroom.github.com/assets/deadline-readme-button-22041afd0340ce965d47ae6ef1cefeee28c7c493a6346c4f15d667ab976d596c.svg)](https://classroom.github.com/a/Z1FNnFtc)

Сервис бронирования переговорных комнат для стажировки в Авито.

## Стек технологий

- **Go 1.22** — язык реализации
- **PostgreSQL 16** — база данных
- **chi** — HTTP-роутер
- **golang-jwt** — работа с JWT
- **Docker Compose** — деплой

## Запуск

```bash
docker-compose up --build
```

Сервис будет доступен на `http://localhost:8080`.

### Makefile

```bash
make up      # Запуск проекта
make down    # Остановка и удаление данных
make seed    # Наполнение БД тестовыми данными
make test    # Запуск тестов
```

## Архитектура

```
cmd/server/main.go          — точка входа
internal/
  config/                    — конфигурация из переменных окружения
  model/                     — доменные модели и типы ошибок
  store/                     — слой работы с БД (PostgreSQL)
  service/                   — бизнес-логика
  handler/                   — HTTP-обработчики
  middleware/                 — JWT-авторизация (контекст claims)
tests/                       — интеграционные тесты
```

Трёхслойная архитектура: **Handler → Service → Store**. Каждый слой отвечает за свою зону ответственности:
- **Handler** — разбор HTTP-запроса, валидация входных данных, формирование ответа
- **Service** — бизнес-правила и валидация
- **Store** — SQL-запросы к PostgreSQL

Для обработчиков (handler) используются интерфейсы сервисов, что позволяет легко подменять зависимости в тестах.

## Проектные решения

### Генерация слотов

Выбран подход **lazy generation** (генерация по запросу):

- При первом запросе списка слотов на конкретную дату система проверяет, есть ли слоты для этой комнаты и даты в БД
- Если нет — генерирует их на основе расписания и вставляет с `ON CONFLICT DO NOTHING`
- Повторные запросы на ту же дату просто читают из БД

**Почему этот подход:**
- Не требует фоновых задач (cron/worker)
- Не создаёт лишних записей для неиспользуемых дат
- Слоты получают стабильные UUID при первом обращении
- Подходит для сценария «99.9% запросов — ближайшие 7 дней»

**Оптимизация производительности (эндпоинт слотов — SLI 200ms):**
- Быстрая проверка через `SELECT COUNT(*)` с индексом — если слоты уже есть, вставка пропускается
- Batch-INSERT одним запросом вместо поштучной вставки
- Индекс `idx_slots_room_start` на `(room_id, start_time)` для быстрого поиска
- Подзапрос `NOT EXISTS` для фильтрации забронированных слотов использует индекс `idx_bookings_slot_status`

### Конкурентный доступ

- **Двойное бронирование** предотвращается partial unique index: `idx_bookings_slot_active ON bookings (slot_id) WHERE status = 'active'` — БД гарантирует, что на один слот может быть только одна активная бронь
- **Двойное создание расписания** защищено UNIQUE constraint на `schedules.room_id` + проверка в сервисном слое
- **Параллельная генерация слотов** безопасна благодаря `ON CONFLICT DO NOTHING`

### Идемпотентная отмена

Повторный вызов `POST /bookings/{id}/cancel` на уже отменённую бронь возвращает `200` с текущим состоянием (`status: cancelled`), а не ошибку.

### Conference Link (дополнительное задание)

При `createConferenceLink: true` система генерирует mock-ссылку формата `https://conference.example.com/<uuid>`. В production вместо мока был бы HTTP-клиент к внешнему сервису.

**Обработка сбоев внешнего сервиса (проектное решение):**
- Если внешний сервис недоступен — бронь создаётся без ссылки, пользователь уведомляется
- Если ошибка после успешного ответа — используется saga/компенсация (отмена ссылки)
- В текущей реализации мок всегда успешен

### Авторизация

- `/dummyLogin` возвращает JWT с фиксированными UUID: `00000000-...01` для admin, `00000000-...02` для user
- JWT содержит `user_id` и `role`, подписан HMAC-SHA256
- Middleware извлекает claims и кладёт в контекст запроса
- `user_id` при создании брони берётся из токена, не из тела запроса

## Тестирование

### Юнит-тесты
```bash
go test ./internal/... -v -cover
```

Покрывают:
- JWT генерация/парсинг (auth_test.go)
- Валидация расписания (schedule_test.go)
- Парсинг времени (slot_test.go)
- Все HTTP-обработчики с mock-сервисами (handler_test.go): авторизация, роли, CRUD, ошибки

### Интеграционные тесты
```bash
INTEGRATION=true go test ./tests/... -v
```

Сценарии:
1. Создание переговорки → расписание → бронирование
2. Отмена бронирования (включая идемпотентность)

## API

Полная спецификация — в файле [api.yaml](api.yaml).

| Метод | Путь | Роль | Описание |
|-------|------|------|----------|
| GET | `/_info` | — | Хелсчек |
| POST | `/dummyLogin` | — | Получить JWT |
| GET | `/rooms/list` | admin, user | Список комнат |
| POST | `/rooms/create` | admin | Создать комнату |
| POST | `/rooms/{id}/schedule/create` | admin | Создать расписание |
| GET | `/rooms/{id}/slots/list?date=` | admin, user | Доступные слоты |
| POST | `/bookings/create` | user | Забронировать |
| GET | `/bookings/list` | admin | Все бронирования |
| GET | `/bookings/my` | user | Мои бронирования |
| POST | `/bookings/{id}/cancel` | user | Отменить бронь |
